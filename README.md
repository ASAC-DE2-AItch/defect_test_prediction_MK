## 1. 이 프로젝트는 무엇인가요?

반도체 공장에서는 웨이퍼(얇은 반도체 원판)를 만들 때, 공정 장비에 부착된 수많은 센서가 온도, 압력, 가스 유량 등의 데이터를 실시간으로 기록합니다. 이것을 **FDC(Fault Detection and Classification) 데이터**라고 합니다.

이 프로젝트는 이러한 센서 데이터를 분석하여 각 웨이퍼의 **불량 검사 결과(C65 값)**를 미리 예측합니다. 쉽게 말해, "이 웨이퍼가 얼마나 불량이 있을지"를 실제 검사 전에 추정하는 것입니다.

### 비유로 이해하기

> 요리에 비유하면, 오븐의 온도, 조리 시간, 재료 상태 등의 기록을 보고 "이 빵이 얼마나 잘 구워질지" 점수를 매기는 것과 같습니다. 실제로 빵을 잘라보지 않아도 조리 과정의 데이터만으로 결과를 예측하는 거죠.

---

## 2. 왜 이런 예측이 필요한가요?

| 문제 | 해결 |
|------|------|
| 불량 검사는 시간과 비용이 많이 듦 | 예측으로 **검사 우선순위**를 정할 수 있음 |
| 불량이 발견되면 이미 늦음 | 센서 데이터로 **사전 경고** 가능 |
| 어떤 공정 조건이 불량을 만드는지 모름 | 중요한 센서를 파악하여 **원인 분석** 가능 |

## 3. 사용하는 데이터

### 3.1 데이터 파일 구성

```
문제1(하)/
├── train_data.csv      ← 학습용 데이터 (정답 포함)
├── valid_X.csv         ← 검증용 입력 데이터
├── valid_Y_problem.csv ← 검증용 제출 양식
├── test_X.csv          ← 최종 평가용 입력 데이터
└── test_Y_problem.csv  ← 최종 평가용 제출 양식

문제1_하_answer/
├── valid_Y_answer.csv  ← 검증용 실제 정답
└── test_Y_answer.csv   ← 최종 평가용 실제 정답
```

### 3.2 데이터가 담고 있는 정보

각 행(row)은 하나의 웨이퍼가 공정의 특정 단계(Step)를 거칠 때 센서가 기록한 값입니다. 하나의 웨이퍼는 여러 단계를 거치므로, **같은 웨이퍼 ID(C64)를 가진 행이 여러 개** 존재합니다.

| 컬럼 종류 | 예시 | 설명 |
|-----------|------|------|
| 웨이퍼 ID | C64 | 웨이퍼를 구분하는 고유 번호 |
| 레시피 | C6 | 어떤 공정 방법(레시피)을 사용했는지 |
| 공정 단계 | C7 | 몇 번째 단계인지 (예: 1단계, 2단계...) |
| 경과 시간 | C41 | 공정이 시작된 후 얼마나 지났는지 |
| PM 시간 | C33 | 장비 정비(PM) 후 경과 시간 |
| FDC 센서 값 | C1, C3, C4, ... | 온도, 압력, 유량 등 센서 측정값 |
| **불량 검사 결과** | **C65** | **예측해야 할 값 (타깃)** |

### 3.3 사용하지 않는 컬럼과 이유

| 컬럼 | 제외 이유 |
|------|-----------|
| C20, C21, C22 (Lot ID) | 특정 생산 묶음 번호라서, 새로운 데이터에는 없는 번호가 나옴 → 일반화 불가 |
| C34, C35, C38 | C64와 중복되는 웨이퍼 식별자 |
| C14, C24 | 모든 행이 같은 값 → 정보 없음 |
| C2, C13 등 6개 | 전부 비어있음 (결측) |
| C26, C28, C29, C37 | 명세상 제외 지정 |

## 4. 분석 파이프라인 (단계별 설명)

`modeling_pipeline.ipynb` 노트북은 아래 9단계로 구성됩니다.

### 단계 1: 데이터 로드

CSV 파일들을 읽어와 메모리에 올립니다.

### 단계 2: 전처리 — 불필요 컬럼 제거

위에서 설명한 "사용하지 않는 컬럼"을 제거하고, 학습/검증/테스트 데이터의 컬럼 순서를 통일합니다.

### 단계 3: Feature Engineering (피처 엔지니어링)

> **핵심 단계입니다.**

원본 데이터는 한 웨이퍼당 여러 행이 있습니다 (공정 단계마다 한 행). 이것을 **한 웨이퍼 = 한 행**으로 요약합니다.

각 센서 값에 대해 아래와 같은 통계량을 계산합니다:

| 통계량 | 의미 | 비유 |
|--------|------|------|
| **평균 (mean)** | 전체 공정 동안의 평균값 | 하루 평균 기온 |
| **표준편차 (std)** | 값이 얼마나 흔들렸는지 | 기온 변동폭 |
| **최솟값 (min)** | 가장 낮은 값 | 최저 기온 |
| **최댓값 (max)** | 가장 높은 값 | 최고 기온 |
| **중앙값 (median)** | 중간값 | 체감 온도 |
| **범위 (range)** | 최댓값 - 최솟값 | 일교차 |
| **변화량 (delta)** | 마지막 값 - 처음 값 | 오늘 vs 어제 기온 차이 |
| **기울기 (slope)** | 시간에 따라 올라가는지 내려가는지 | 기온이 계속 오르는 추세인지 |

추가로 다음 정보도 포함합니다:
- **공정 단계 수 (n_rows)**: 이 웨이퍼가 거친 단계 수
- **총 소요시간 (C41_total)**: 전체 공정에 걸린 시간
- **PM 시간 (C33)**: 장비 정비 후 얼마나 지났는지
- **레시피(C6) 분포**: 어떤 레시피를 얼마나 사용했는지
- **단계(C7) 분포**: 어떤 단계를 얼마나 거쳤는지

### 단계 4: 피처/타깃 분리

- **피처(X)**: 예측에 사용할 입력 데이터 (센서 통계량들)
- **타깃(Y)**: 예측하고 싶은 값 (C65, 불량 검사 결과)
- **베이스라인**: 가장 단순한 예측 방법(전체 평균)의 RMSE를 먼저 계산하여 기준점으로 삼습니다.

### 단계 5: 모델 학습 (LightGBM + GroupKFold)

#### LightGBM이란?

"결정 트리"를 여러 개 순차적으로 만들어, 이전 트리가 틀린 부분을 다음 트리가 보완하는 방식의 머신러닝 알고리즘입니다. 표 형태의 데이터에서 매우 뛰어난 성능을 보입니다.

#### GroupKFold란?

학습 데이터를 5등분하여, 4개로 학습하고 1개로 검증하는 과정을 5번 반복합니다. 이때 **같은 웨이퍼의 데이터는 반드시 같은 그룹에 속하도록** 하여, 데이터가 새는 것(data leakage)을 방지합니다.

```
[Fold 1] 학습: ████████████████  검증: ████  → RMSE 계산
[Fold 2] 학습: ████████████████  검증: ████  → RMSE 계산
[Fold 3] 학습: ████████████████  검증: ████  → RMSE 계산
[Fold 4] 학습: ████████████████  검증: ████  → RMSE 계산
[Fold 5] 학습: ████████████████  검증: ████  → RMSE 계산
                                    평균 → 최종 CV 점수
```

#### 주요 하이퍼파라미터

| 파라미터 | 값 | 의미 |
|----------|-----|------|
| learning_rate | 0.05 | 각 트리가 배우는 속도 (작을수록 신중) |
| num_leaves | 63 | 트리의 복잡도 (클수록 복잡) |
| n_estimators | 2000 | 최대 트리 개수 |
| early_stopping | 100 | 100번 연속 개선 없으면 멈춤 |
| subsample | 0.8 | 매 트리마다 데이터 80%만 랜덤 사용 |
| colsample_bytree | 0.8 | 매 트리마다 피처 80%만 랜덤 사용 |

### 단계 6: 성능 평가

학습된 모델의 예측값과 실제 정답을 비교합니다.

- **RMSE (Root Mean Squared Error)**: 예측이 실제값에서 평균적으로 얼마나 벗어나는지를 나타냅니다. **낮을수록 좋습니다.**

```
RMSE 계산 방법:
1. 각 예측값과 실제값의 차이를 구한다
2. 차이를 제곱한다 (음수를 없애기 위해)
3. 평균을 낸다
4. 제곱근을 씌운다 (원래 단위로 되돌리기 위해)
```

### 단계 7: Feature Importance (피처 중요도)

어떤 센서(피처)가 불량 예측에 가장 중요한 영향을 미치는지 막대 그래프로 시각화합니다. 이를 통해 공정 엔지니어가 **어떤 센서 값을 주의 깊게 관리해야 하는지** 파악할 수 있습니다.

### 단계 8: 제출 파일 생성

최종 예측 결과를 CSV 파일로 저장합니다:
- `valid_Y_submit.csv` — 검증용 예측 결과
- `test_Y_submit.csv` — 최종 평가용 예측 결과

### 단계 9: 일반화 점검

예측 분포와 실제 분포를 비교하는 그래프를 그려, **모델이 특정 영역에서만 잘 맞추는 건 아닌지** 확인합니다.

## 8. 주요 용어 사전

| 용어 | 설명 |
|------|------|
| **FDC** | Fault Detection and Classification. 공정 장비의 센서 데이터를 수집하는 시스템 |
| **웨이퍼 (Wafer)** | 반도체 칩을 만드는 얇은 원형 기판. 하나의 웨이퍼에서 수백 개의 칩이 만들어짐 |
| **PM (Preventive Maintenance)** | 장비의 예방 정비. PM 후 시간이 지나면 장비 상태가 변할 수 있음 (drift) |
| **레시피 (Recipe)** | 공정 장비에 설정하는 조리법. 온도, 압력, 시간 등의 설정 조합 |
| **RMSE** | Root Mean Squared Error. 예측 오차의 크기. 작을수록 정확한 예측 |
| **피처 (Feature)** | 모델이 예측에 사용하는 입력 변수. 여기서는 센서 통계량 |
| **타깃 (Target)** | 예측하고자 하는 값. 여기서는 C65 (불량 검사 결과) |
| **과적합 (Overfitting)** | 학습 데이터에만 잘 맞고 새 데이터에는 못 맞추는 현상 |
| **교차검증 (Cross-Validation)** | 데이터를 나눠 학습과 검증을 반복하여 모델 안정성을 확인하는 방법 |
| **GroupKFold** | 같은 그룹(웨이퍼)의 데이터가 학습과 검증에 동시에 들어가지 않도록 하는 교차검증 |
| **LightGBM** | 빠르고 정확한 트리 기반 머신러닝 알고리즘 |
| **Feature Importance** | 어떤 피처가 예측에 가장 큰 영향을 미치는지 나타내는 지표 |
| **베이스라인 (Baseline)** | 비교 기준이 되는 가장 단순한 예측 (예: 전체 평균) |
