# 피처 명세서 (Feature Specification) — 핸드오프용

> **목적:** 이 모델이 어떤 **기준으로** 피처를 만들고 어떻게 **가공**했는지를 전달.
> **한 줄 요약:** 원본은 `(웨이퍼 × Step)` 다중 행 → **웨이퍼 1줄로 집계** → 시간/레짐 파생 → **최종 10개만 사용**.
> ⚠️ **타겟:** `C65` = 불량 비트 수 → **낮을수록 좋음**.

---

## 0. 파이프라인 한눈에

```
원본 CSV (WF × Step, 약 123k행)
   │  preprocessing.py
   ▼
train_clean.csv (컬럼 23개 제거 + datetime 파싱 + 시간정렬)
   │  feature_engineering.py
   ▼
wf_features.csv (WF 1줄 × ~589 피처)      ← 여기까지가 "생성"
   │  train.py + outputs/feature_importance.csv
   ▼
실제 모델 입력 = gain 상위 10개 피처만       ← "선택"
```

**핵심 구분:** 피처는 ~589개가 **생성**되지만, 모델이 실제 **사용**하는 건 10개뿐이다(§5). 나머지는 중요도(gain) 상위 선별에서 탈락한다.

---

## 1. 데이터 구조와 집계 단위 (왜 집계하나)

- **원본 1행 = 웨이퍼(WF)의 한 Step**. 한 WF가 여러 Step 행으로 흩어져 있다.
- **예측 단위는 WF 1장**이므로, 모든 피처는 "WF 한 줄로 접는" **집계의 결과물**이다.
- 집계 키: `C64`(WF ID), `C7`(Step 번호).

| 항목 | 컬럼 | 역할 |
|------|------|------|
| WF ID | `C64` | 집계 기준 (모델 입력엔 제외) |
| Step 번호 | `C7` | 집계 기준 (Step별 통계 생성용) |
| 타겟 | `C65` | 예측 대상 (WF당 1값) |
| 정렬/시간 | `C40` | 타임스탬프 (datetime 파싱 + 정렬 + 시간피처 원천) |

## 2. 전처리 (`preprocessing.py`) — 어떻게 정리했나

train / valid / test **동일 로직** 적용. 3단계:

1. **`C40` 문자열 → datetime 변환** (`%Y-%m-%d %H:%M:%S.%f`)
2. **불필요 컬럼 23개 제거** — 아래 3가지 기준
3. **시간순 정렬** (train/valid 분할 시 미래 누수 방지)

### 제거 기준 (23개)

| 기준 | 개수 | 컬럼 |
|------|------|------|
| 완전 결측 (100% 빈 값) | 8 | C2, C13, C26, C37, C43, C47, C53, C55 |
| 상수값 (분산 0, 정보 없음) | 12 | C3, C8, C14, C19, C21, C24, C28, C29, C30, C44, C45, C51 |
| 완전 중복 (다른 컬럼과 100% 동일) | 3 | C36(=C7), C35(=C34), C38(=C64) |

## 3. 피처 생성 **기준** (왜 이렇게 만들었나) — 5대 원칙

이 프로젝트의 모든 피처 결정을 관통하는 원칙. 인수자가 이걸 알아야 "왜 이 피처만 남았는지"가 이해된다.

| # | 원칙 | 의미 | 근거 |
|---|------|------|------|
| **P1** | **프로덕션 안전** | 하드코딩 날짜 금지. PM 날짜는 `pm_log.json`에 추가만 하면 대응. 레시피는 ID로 판별. | 실시간 운영에서 코드 수정 없이 동작해야 함 |
| **P2** | **레짐(regime) 인식** | PM 전후로 C65 레벨이 2배 차이(638→1123) → **시간/레짐 피처가 센서보다 지배적** | 데이터의 실제 구조 (센서→불량 설명력 6.9%뿐) |
| **P3** | **다중공선성 제거** | 상관 0.8+ 로 묶인 그룹에서 **대표만 남김** | 중복 정보 제거 (아래 표) |
| **P4** | **confound 제거** | 상관이 높아도 "레짐/시간의 재인코딩"이면 원값 대신 **플래그로 대체** | C12(r=0.60)가 대표 사례 → `is_special_recipe`로 교체 |
| **P5** | **검증된 것만** | 모든 피처는 **CV(5-fold)로 검증**. 단일 valid 개선(±0.1)·CV 노이즈(±0.22) 이하는 채택 안 함 | 0.3 RMSE 미만 변화는 노이즈로 간주 |

### P3 상세 — 다중공선성으로 제거된 FDC 센서 7개

| 그룹 | 상관 | 남긴 것 | 제거한 것 |
|------|------|---------|-----------|
| group1 [C1,C9,C17,C63] | 0.6~0.9 | C17, C63 | C1, C9 |
| group2 [C4,C5] | 0.95 | C4 | C5 |
| group3 [C11,C15,C16,C31,C62] | 0.6~0.9 | C62 | C11, C15, C16, C31 |
| group4 [C46,C48] | 0.81 | **둘 다 유지** (혼재변수, 다중공선성 아님) | - |

→ FDC 센서 30개 − 7개 = **23개**가 집계 대상.

## 4. 피처 **가공** 방법 (어떻게 만들었나) — `feature_engineering.py`

### 4-1. FDC 센서 집계 (2단계 pivot)

**대상:** FDC 23개 중 수치형 22개 + 문자열 1개(`C6`).

**[1단계]** `groupby(WF, Step)` → 5가지 통계 집계
- 통계: **mean, std, max, min, last** (`AGG_FUNCS`)
- 기준: EDA에서 mean보다 max/min이 강한 신호를 준 경우가 많아 5종 전부 생성 후 중요도로 선별
- **C41**(Step 내 경과시간) → Step별 `max` = **Step 소요시간** 대리지표 (`C41_max`)

**[2단계]** `pivot` → WF 한 줄로 펼침
- 컬럼명 형식: **`{센서}_{통계}_step{번호}`** (예: `C17_mean_step1`, `C60_std_step4`)
- 해당 WF에 없는 Step 칸 = **NaN** (LightGBM이 자체 분기 처리, 별도 대치 안 함)

**문자열 예외:** `C6`(레시피 코드)는 수치 집계 불가 → WF 내 고정값이라 `first()`로 추출.

### 4-2. 시간 파생 피처 (P1·P2 반영)

원본 타임스탬프(`C40`)와 PM 이벤트 로그(`pm_log.json`)로 계산:

| 피처 | 계산식 | 설명 |
|------|--------|------|
| `days_since_last_pm` | PM 이후 WF: `(측정일 − 최근 PM일)`<br>PM 이전 WF: `(측정일 − 2018-12-01)` | 최근 PM부터 경과일. **PM마다 0으로 리셋** → 복수 PM 주기 지원 |
| `pm_cycle_count` | 지금까지 발생한 PM 횟수 | pre=0, 1차 PM 후=1, … (메타데이터로 유지, 모델엔 직접 미사용) |
| `hour` | `측정 시각.hour` (0~23) | 3교대 근무·장비 열적 안정성 |
| `is_post_pm` | `(pm_cycle_count > 0)` → 0/1 | **레짐 플래그.** 숫자 count 대신 플래그를 쓰는 이유: cycle 3 예측 시 미학습 값(3)이 나오지 않도록 (P1) |

> **왜 `pm_cycle_count`(숫자)를 모델에 안 쓰나:** "몇 번째 cycle인지"는 R2R/재학습이 다루는 **외부 상태**다. 모델엔 레짐 구분(pre/post)만 플래그로 주어 cycle 간 교환 가능하게 한다.

### 4-3. 교호(interaction) 피처 3개

시간축 상호작용이 C65에 강한 신호를 준다(실험으로 −4.6 RMSE 검증). 구성요소가 전부 프로덕션-안전이라 PM 반복에도 확장됨.

| 피처 | 계산식 | 의미 |
|------|--------|------|
| `dslp_x_hour` | `days_since_last_pm × hour` | PM 이후 경과에 따라 3교대 패턴 효과가 달라지는 정도 |
| `hour_x_c33` | `hour × C33` | 장비 시간 카운터 상태 × 측정 시각 복합 효과 |
| `post_pm_days` | `days_since_last_pm × (pm_cycle_count > 0)` | PM 이후 경과일만 남기는 **마스크** (pre-PM=0) |

> ⚠️ **`post_pm_days`는 곱셈이 아니라 마스크다.** `× pm_cycle_count`(곱)로 하면 PM 2회차(count=2)에서 경과일이 2배로 왜곡된다. `× (count>0)`(게이팅)이라 PM이 반복돼도 경과일이 그대로 유지된다 (P1).

### 4-4. 레시피 플래그 (P4 반영)

| 피처 | 계산식 | 설명 |
|------|--------|------|
| `is_special_recipe` | `(C6 == 'C6_1')` → 0/1 | 특수 레시피(C6_1) 여부. 이 레시피는 C65가 ~280 낮은 **진짜 신호**(우수 배치) |

> **왜 C12 원값을 안 쓰고 플래그로 대체했나(P4):** C12(Vdc setpoint)는 겉보기 상관 r=0.60이지만 대부분 **PM 레짐 confound**(레짐 내부 상관 ≈0). 원값을 쓰면 노이즈가 낀다. 특수 레시피의 진짜 신호만 이진 플래그로 뽑아냈다 (valid 39.25→38.34). C6는 레시피 ID라 **실시간에 항상 아는 값** → 안전. 신규 레시피(C6_2 등)는 자동으로 0(주력) 폴백.

## 5. 최종 사용 피처 **10개** (실제 모델 입력)

`outputs/feature_importance.csv`의 gain 상위 10개. 시간·레짐 축 7개 + 센서 3개 구성.

| 순위 | 피처 | 출처/생성식 | 종류 | 프로덕션 안전성 |
|------|------|------------|------|----------------|
| 1 | `is_post_pm` | `(pm_cycle_count>0)` | 레짐 플래그 | ✅ pm_log 기반 |
| 2 | `post_pm_days` | `days_since_last_pm × (count>0)` | 교호(마스크) | ✅ |
| 3 | `days_since_last_pm` | `측정일 − 최근 PM일` | 시간 | ✅ |
| 4 | `C33` | 원본(FDC 장비 시간 카운터) | 메타 | ✅ 장비 제공값 |
| 5 | `dslp_x_hour` | `days_since_last_pm × hour` | 교호 | ✅ |
| 6 | `hour` | `측정시각.hour` | 시간 | ✅ |
| 7 | `hour_x_c33` | `hour × C33` | 교호 | ✅ |
| 8 | `C60_mean_step4` | FDC 집계(§4-1) | 센서 | ✅ |
| 9 | `C59_mean_step4` | FDC 집계(§4-1) | 센서 | ✅ |
| 10 | `is_special_recipe` | `(C6=='C6_1')` | 레시피 플래그 | ✅ 레시피 ID |

**센서 블록 주의:** C59/C60은 gain이 상대적으로 낮아 자동 top-10 선별에선 탈락하기 쉽지만, **집단적으로 필수**(3종 전부 빼면 CV 46으로 폭망)라 유지한다. C60이 유일한 진짜 within-cycle 센서, C59는 CV-중립 잉여. (`C60_std_step4`는 C60 통계 중복이라 ablation으로 제거됨.)

## 6. 피처 **선택** 방법

1. 1차 학습: `feature_importance.csv`가 없으면 **전체 수치 피처**로 학습 → gain 기준 중요도 저장.
2. 2차 학습: `train.py`가 `feature_importance.csv`에서 **`gain > 0` 상위 `TOP_N_FEATURES`(=10)** 만 선택해 재학습.
3. 검증: 모든 미세 결정은 **5-fold CV**로 확인 (P5). 상관계수만으로 판단하지 않고 **ablation + 레짐 내부 상관**으로 교차검증.

> `TOP_N_FEATURES` 값(train.py)을 바꾸면 다른 조합이 선택된다. 현재 10이 스윗스팟(train/valid gap 관리).

## 부록 A. 기각된 파생 피처 (재시도 방지)

| 피처 | 기각 이유 |
|------|----------|
| `C33_log`, `C33_inv` | LightGBM 트리에서 단조변환은 동일 정보 (gain=0) |
| `C34`(슬롯 번호) | 추가 시 +0.22 악화 (노이즈) |
| `days_from_start` | 캠페인 시작일 하드코딩 → 프로덕션 부적합 (P1 위반) |
| `after_pm_shift`, `days_since_pm` | 단일 PM 날짜 하드코딩 → 시간피처로 대체 (P1) |
| `dayofweek` | 수요일 클러스터가 PM+2일과 겹치는 confound (P4) |
| Vdc 편차/센서×센서 교호 | 전부 CV 악화 — C65는 시간축 지배라 센서 교호 무효 |
