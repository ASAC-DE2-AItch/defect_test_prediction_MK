## 0. 요약 (TL;DR)

**무엇을 하나.** 지금까지의 모든 버전(v1~v7)은 시간 컬럼(C10/C39/C40/C41)을 전부 버리고 센서 집계만으로 학습했다. FEATURE_SPEC이 전달한 외부 모델은 정반대로 **시간/PM 레짐 피처 7개 + 센서 3개**만으로 valid RMSE ~38.3을 기록했다고 명세한다. v8은 이 레짐 피처 체계를 우리 파이프라인에 이식·재현하고(M0), 그 위에 v5의 유효 요소(C23 타깃인코딩, row-level 프레임, WF context)를 하나씩 얹어 ablation으로 최종 조합을 확정한다. 피처셋이 확정되면 LightGBM 외 모델 후보 7종(XGBoost·CatBoost·HistGB·배깅 트리·정칙화 선형·레짐 세그먼트 모델)을 동일 조건에서 벤치마크해 최종 모델을 데이터로 선정하고(M5), 마지막에는 train+valid+test를 병합해 **날짜 기준으로 재분할**하는 미래 예측 시뮬레이션(M8)으로 "새 trace 데이터가 들어오는 운영 상황"의 대응력을 검증한다.

**왜 되는가 (이번 기획 단계에서 실데이터로 사전 검증 완료).**

1. 캠페인 도중 **PM 1회가 실재**한다: 2018-12-23 08:32(C33=39) → 2018-12-24 01:31(C33=1)로 C33 리셋 + ~17시간 생산 공백.
2. PM 전후로 C65 레벨이 **637.5±40.1 → 1123.5±159.4로 2배 점프** — FEATURE_SPEC의 주장(638→1123)과 정확히 일치.
3. 지금까지 "단일 최강 센서"였던 C17(전체 상관 −0.797)은 레짐 내부에서 **pre −0.19 / post +0.10으로 붕괴** — 즉 v1~v7은 레짐(시간)을 센서로 간접 추정하느라 ~60에 갇혀 있었다. 레짐을 직접 주면 천장이 사라진다.
4. valid/test도 train과 동일 기간을 커버(pre-PM 비율 30.4~30.5%로 train 30.9%와 동일) → 레짐 피처가 대회 평가에 그대로 적용 가능.
5. (v1.2) FEATURE_SPEC 작성자의 **원본 파이프라인·학습 모델을 확보**(`pm_feature/`): 원본 pm_log.json이 우리 재구성과 일치하고, 원본 코드 주석으로 기대 성능이 **CV ~41 / valid ~38.3**으로 확정, valid도 동일한 `valid_Y_answer.csv` 기준임을 확인 — M0는 "명세 해석"이 아닌 **코드 이식**이 됐다.

**성공 기준**: 대회 트랙 valid·test RMSE **≤ 40** (스트레치 ≤ 38) + 실무 트랙 Lot-CV 갭 축소(v5 +7.8pt → ≤ +3pt 목표) + v5의 최대 오차 구간(정답 742~976, 편향 +58.8) 해소 + 시간 재분할 미래 예측에서 대회 대비 열화 ≤ +10pt(잠정, G5).

**전체 흐름**:

```
Phase 0 준비 → Phase 1 검증 EDA(02_EDA_regime.ipynb, pm_log.json 생성)

## 1. 배경 — 왜 v8인가

### 1.1 두 재료의 현재 상태

| 축 | **modeling_v5 (우리 최선)** | **FEATURE_SPEC 모델 (핸드오프 명세)** |
|---|---|---|
| 예측 단위 | row(step) 단위 예측 → WF 평균 | WF 1줄 집계 |
| 시간 피처 | **없음** (C10/C39/C40/C41 전부 drop) | **핵심** (days_since_last_pm, is_post_pm, hour, 교호 3종) |
| 센서 피처 | 원본 22종 + WF 집계 broadcast (~185개) | C33 + C59/C60_mean_step4 + is_special_recipe (gain 상위 10개만) |
| C23 (28종 레시피) | OOF 타깃인코딩 사용 (유효 확인) | 미사용 |
| 다중공선성 | 미처리 (트리에 위임) | 상관 0.8+ 그룹 대표만 (30→23 센서) |
| CV | GroupKFold(C64) 5-fold | 5-fold CV (그룹 기준 미상) |
| 성능 | CV 62.63 / valid 61.38 / test 60.52 | **valid ~38.3** (명세 §4-4 기준) |
| 검증 위생 | 누수 검증 완료 (Lot-CV 70.42, 낙관 +7.8pt) | 명세만 있고 코드·재현물 없음 |

**상보성이 완벽하다.** v5는 "무엇을 어떻게 검증할지"의 절차가 강하고, FEATURE_SPEC은 "무엇을 피처로 쓸지"의 내용이 강하다. v8 = FEATURE_SPEC의 피처 내용 × v5의 검증 절차.

### 1.2 v1~v7이 60 천장이었던 진짜 이유 (이번에 규명됨)

지금까지의 진단은 "tabular 표현력 소진"이었다. 이번 기획 검증으로 원인이 다르게 확인됐다.

1. **타깃의 지배 구조는 PM 레짐이다.** C65 분산의 대부분이 "PM 전이냐 후냐 + PM 후 며칠 지났냐"로 설명된다 (pre std 40.1 vs 전체 std 262).
2. **v1~v7은 그 레짐 정보를 입력에서 제거했다.** C10/C39(epoch 초 단위 절대시각, C40과 동일 정보)는 EDA에서 상관 +0.478로 잡혔음에도 "중복 시간 변수"로만 취급됐고, v5에서 최종 drop됐다. C40의 "step 간격 3초 균일" 판정은 **WF 내부** 변동에 대한 것이었고, **WF 간(캠페인 축) 정보**는 아무도 쓰지 않았다.
3. **모델은 센서로 레짐을 간접 복원하며 버텼다.** C17(-0.797), C11_wf_min(v5 gain 1위), C33 등은 레짐과 함께 드리프트하는 **레짐 프록시**였다. 프록시는 노이즈가 커서 ~60이 한계였다.
4. 그래서 v5의 최대 오차가 **레짐 경계 구간(정답 742~976, 편향 +58.8)** 에 집중됐다 — pre(≈640) 레벨과 post(≈1120) 레벨 사이의 wafer를 프록시로는 구분 못 한 것.

### 5.2 pm_log 형식 (v1.5 갱신 — verdict 필드 도입)

정본 `pm_log.json`(루트)은 **verdict 포맷**을 사용한다: `[{"date": "2018-12-24", "type": "major", "verdict": "loud"}]` — v1.4의 타입 포맷에 **Qual 요란 판정**(verdict: `loud`/`quiet`) 필드를 추가했다(팀 크로스체크 회신). type(정비 내용: major/minor)과 verdict(이벤트 판정: 레짐이 실제로 바뀌었는가)는 **다른 축**이며, **피처는 verdict만 소비한다** — type은 기록·해석용으로 유지. 이식한 `_compute_time_features()`가 이 포맷을 소비하도록 수정하되, 신규 PM 발생 시 **엔트리 1개 추가로 끝난다**는 성질(P1)은 유지한다 (quiet → 레짐 상태 유지, loud → 상태 갱신 — §7.1).

**verdict 기입 규칙 (누수 방어 — "C65 역추정 금지"의 verdict 확장)**: verdict는 **PM 직후 D+0 Qual 판정으로 고정**하고, 생산 wafer의 C65를 보고 **사후 수정하지 않는다** (사후 수정 = 타깃 정보의 피처 유입). 데이터 내 12-24의 `verdict="loud"`는 당시 Qual이 봤을 X 채널 신호(C17 델타 등)와 정합하며, C65 급등은 사후 정합 확인용이었다는 기존 입장과 동일 구조다. 탐지 근거(규칙·시각·증거·type·verdict·3셋 교차검증)는 `pm_log_meta.json`으로 분리 저장한다:

  "verified_on": ["train", "valid_X", "test_X"]
}
```

주의 2가지. (1) `pm_date_used`는 **first_post_wafer_ts 기준**으로 잡는다(경계 안전: 공백 구간엔 wafer가 없으므로 어느 쪽이든 오분류 0이지만, 규칙은 하나로 고정). (2) valid/test에서도 같은 규칙으로 **독립 탐지**해 train과 동일한 pm_date가 나오는지 assert한다 — 이것이 "레짐 피처는 X만으로 계산 가능(실시간 안전, P1)"의 증명이 된다.

### 5.3 Phase 1 종료 판정 (G0)

E1·E2·E3 통과 + pm_log.json 생성 + `02_EDA_regime` 리포트 셀(판정 요약표 출력)까지. **여기서 go가 나와야 Phase 3 진입.**

## 8. Phase 4 — 모델링 실험 매트릭스

### 8.0 공통 프로토콜

1. **한 실험 = 한 변경.** 여러 변경 동시 적용 금지 (v4 실패 교훈).
2. **판정 기준**: GroupKFold(C64) 5-fold CV OOF(WF-level RMSE)가 1차 기준. **|ΔCV| < 0.3은 노이즈로 간주, 채택 안 함** (FS P5). valid는 참고, test는 마일스톤에서만 조회.
3. 모델: **M0~M4는 LightGBM 고정** — 피처 효과를 모델 효과와 분리해 판정하기 위함 (소표본 tabular에서의 GBDT 우위는 v6 결론). 파라미터는 **기본값+early stopping** 통일 (v5 수동값 lr=0.03, num_leaves=127을 시작점으로 사용 가능하되 전 실험 동일 유지). **다른 모델 후보의 검증은 피처셋 확정(M4) 이후 M5에서** 동일 조건으로 수행한다 (§8.3).
4. 모든 실험은 `run_experiment()` 러너(부록 A Cell 7)로 실행해 results에 자동 기록.
5. 시드 42 고정, `n_jobs=-1`, early_stopping(100).

### 8.1 실험 정의

### 8.2 실행 순서와 중단 규칙

```
M0 ──G1──▶ M1 ─▶ M2 ─▶ M3 ─▶ M4 ──G2──▶ M5a 모델 벤치 ─▶ M5b 튜닝 ─▶ (M6) ─▶ M7 ──G3──▶ Phase 6
 │ 실패                                   │ 실패                                   │
 ▼                                        ▼                                       ▼ (대회 제출물 확정 후)
 R1 절차 (피처식 대조 → MK 원본 요청)        M0 구성 롤백 후 M5                      M8 ──G5──▶ 종료
```

M1~M3은 독립 가설이므로 순서 교체 가능하나, **M4(피처 선택) → M5a(모델 선택) → M5b(튜닝)는 반드시 이 순서** — 순서를 바꾸면 선택과 튜닝이 서로를 무효화한다. M8은 valid/test 정답을 학습에 쓰는 유일한 절차이므로 **모든 대회 산출물 확정 후 맨 마지막**에만 실행한다 (§9.4 방화벽).

> **진행 현황 (v1.6, 2026-07-10)**: M0 ✅ G1 완결 (M0a 37.97 / M0b CV 40.35·valid 38.40·test 38.91, M-T 44.26 → 센서 +3.91pt) → M1 ❌ 기각(+0.68, C23_te=레짐 프록시) → M2 ❌ 기각(최선 TOP_15 +3.59, probe+3 +0.71 — **§8.5 관찰 트랙의 원료**) → M3 ❌ 기각(+10.52, row-level 열세) → M4 ✅ **G2 통과, 코어 10 동결** (레짐 제거 +218.9 / 센서 제거 +3.91 / dedup 기각 +8.01 / gain-greedy 열세 3.6pt). **다음: M5a** (3트랙 병행, §8.5). 상세는 `REPORT/modeling_v8_REPORT_01~05`.

### 8.4 센서 피처 보존 원칙 (v1.3 — "시간 피처만 남는 모델" 방지)

**우려**: 레짐/시간 피처가 gain을 독식해 센서 피처가 선별에서 전멸하면, v8은 드리프트 일정표 모델이 되어 v5(센서 기반)를 합치는 의미가 사라진다.

**데이터가 말하는 예상 — 전멸하지 않는다.** 원본의 자체 ablation이 근거: top-10에서 센서 블록(C59/C60_mean_step4 등 3종)을 전부 제거하면 CV가 41→46으로 ~5pt 폭락한다 (FEATURE_SPEC §5 "집단적으로 필수"). gain **순위표**에서 시간 피처가 상위를 독식하는 것과 센서 **블록의 기여가 사라지는 것**은 다른 문제다. C60_mean_step4는 원본이 "유일한 진짜 within-cycle 센서"로 명시한 피처이고, E7 confound 스캔이 이런 "레짐 통제 후에도 신호가 남는 센서"를 추가 발굴한다.

**역할 분해 — v5가 사는 자리.** 시간/레짐 피처는 "이 시점의 기대 수준(level)"을 맞히고, 센서 피처는 "같은 시점의 다른 wafer 대비 이 wafer의 편차(wafer별 판별)"를 맞힌다. 실무 목적(어떤 wafer를 우선 검사할지)의 본질은 후자다 — **시간-only 모델은 일정표일 뿐 wafer 판별력이 0**이다. 신뢰도 계층·이상 wafer 플래깅(Phase 6)도 센서 신호 위에서만 의미가 있다. v5의 센서 작업(집계 풀·검증 규율)이 v8에서 담당하는 부분이 정확히 이 within-regime 판별이다.

**개수 비율 "7:3"은 과대 표현이다 — 시간 블록의 실질 자유도는 ~3 (v1.4).** 그룹 A 7종 중 `days_since_last_pm`·`C33`·`high_regime_days`는 서로 준중복이고(C33 ≈ 1.62 × 경과일의 장비 카운터 버전, high_regime_days는 dslp의 마스크), 교호 2종(`dslp_x_hour`·`hour_x_c33`)은 파생이다. 독립 신호는 **{레짐 스위치, 주기 내 경과일, 시각} 3개**뿐이며, 트리는 같은 정보를 여러 인코딩에 나눠 쓸 뿐이다. 따라서 비율은 실험에서 다음 경로로 조절된다: ① **시간 블록 dedup** (M4 서브스텝) — 최소 시간셋 3종 vs 전체 7종을 비교해 |ΔCV| < 0.3이면 최소셋 채택 → 성능 손실 없이 "시간 3 : 센서 3+α"에서 출발. ② **M2 + TOP_N 확장(12~15)** — 시간 정보는 3~4개에서 포화하므로 늘어나는 슬롯은 사실상 전부 센서 몫 (E7이 발굴한 C25 급등 내부 +0.459가 첫 후보). ③ **2단계 잔차 모델** (아래 장치 4) — 구조적으로 시간=1단, 센서=2단. 단, **비율 자체를 목표로 삼지 않는다** (P5): 조절은 CV 무손실 범위에서만 하고, 시간/센서의 몫은 개수가 아니라 ablation 기여로 보고한다.

> **v1.6 실측 후기 (M2·M4 결과)**: 이 절의 가설 중 **①(dedup)과 ②(TOP_N 확장)는 실측 기각됐다** — 최소 시간셋 3종은 +8.01pt 악화(나머지 4종이 집단으로 ~8pt 기여 — "자유도 ~3" 가설이 이 데이터에선 불성립), TOP_N 확장·C25 투입도 전부 악화(M2). 반면 **센서 보존 자체는 확증** — 블록 ablation +3.91pt(제거 시 악화)로 유지 규칙을 압도적으로 만족, C59/C60의 "저-gain·고-집단가치" 경고(FS §5)도 재현. ③(잔차 2단계)만 미실험 옵션으로 남는다. 기각된 두 후보 구성은 폐기하지 않고 §8.5 관찰 트랙(F-T15·F-P3)으로 동결해 계속 추적한다.

**보존 장치 4가지**:

1. **M-T 시간-only 대조군 상시 유지** — 모든 마일스톤에서 "센서 증분 = 현재 모델 − M-T"를 기록. 증분이 노이즈 수준으로 수렴하면 그 사실도 숨기지 않고 보고한다 (그 자체가 발견).
2. **블록 ablation 채택 규칙** (M4) — 센서는 개별 gain 순위가 아니라 블록 단위 제거 실험(ΔCV ≥ +0.3)으로 유지를 판정. 자동 top-N 선별이 센서를 떨어뜨려도 블록 검증을 통과하면 수동 복원한다 (원본이 C59/C60에 했던 것과 동일).
3. **E7 우선 투입** — M2 센서 풀 확장 시, confound 스캔에서 within-regime 신호가 확인된 센서를 우선 후보로 명시 (시간 프록시성 센서에 gain을 뺏기지 않도록).
4. **(그룹 D 후보) 2단계 잔차 모델** — 1단: 시간/레짐 피처로 기대 수준 예측 → 2단: v5 스타일 센서 피처로 **잔차** 학습. 센서의 역할이 구조적으로 보존·해석되는 변형(잔차 = wafer 고유 이상 신호). 단일 모델과 성능이 동급이면 실무 설명용으로 이쪽을 채택할 수 있다. M5 벤치 이후 옵션으로 1회 실험.

**REPORT 의무 항목**: 3각 비교표 — **v5(센서-only, valid 61.4) / M-T(시간-only) / v8(결합)** — 와 블록별 기여 분해(시간·센서·레시피)를 최종 보고에 반드시 수록해, "v8에서 센서(=v5의 유산)가 몇 pt를 담당하는가"를 숫자로 남긴다.

### 8.5 관찰 트랙 (v1.6 신설) — 기각 후보 2종의 병행 성능 추적

**배경·목적.** M4가 코어 10을 G2로 동결했지만, 그 우위는 지금까지 **LightGBM(M8_PARAMS) × 랜덤 wafer CV** 한 조합 위에서만 확인됐다. 사용자 지시(2026-07-10)에 따라 M2의 기각 후보 2종을 관찰 트랙으로 동결하고, 남은 전 단계(M5a/M5b/M7/M8)에서 챔피언과 나란히 성능을 계속 측정한다. 목적 3가지: ① **모델 특이성 검증** — 다른 학습기(ordered boosting, 정칙화 선형+스플라인)에서도 코어 10이 이기는가. ② **검증 스킴 특이성 검증** — Lot-CV·시간 재분할에서도 순위가 유지되는가. 특히 **M8-S2(COLD: 레짐 피처가 학습에서 전부 상수)**에서는 레짐 변화를 감지할 수 있는 유일한 채널이 센서 프록시(C12 계열 — C17 동형)이므로, 프록시를 품은 트랙이 raw bias에서 F-C10보다 완만할 수 있다는 가설(**H-T1**)을 명시적으로 검정한다 — 확인되면 "pm_log 공백/레짐 미기록 상황의 백업 피처셋"이라는 실무 가치가 생긴다. ③ **M6 앙상블 다양성 원료** — 피처셋 변형은 모델 변형과 같은 오차 탈상관 축.

**트랙 정의 (M2 스냅샷 기준 동결 — 재유도 금지, 목록 상수화)**:

| 트랙 | 구성 | 피처 수 | 기준 성적 (LGBM·M8_PARAMS, M2) |
|---|---|:---:|---|
| **F-C10** (챔피언) | 코어 10 (G2 동결) | 10 | CV **40.35** / valid 38.40 / test 38.91 |
| **F-T15** | M2 pass1 gain TOP_15: `is_high_regime, high_regime_days, days_since_last_pm, C12_mean_step6, C33, dslp_x_hour, C12_mean_step7, hour, C4_mean_step4, C4_max_step1, hour_x_c33, C12_max_step6, C61_mean_step1, C60_std_step4, C62_mean_step1` | 15 | CV 43.94 / valid 41.67 / test 43.25 |
| **F-P3** | 코어 10 + `{C12_mean_step6, C12_mean_step7, C4_mean_step4}` | 13 | CV 41.05 (valid/test 미조회 — M5a에서 최초 기록) |

주의: F-T15에는 **코어 센서 `C59/C60_mean_step4`·`is_special_recipe`가 없다**(gain ~18위권 탈락) — "집단 필수 센서를 잃은 자동선택 셋"의 대표. F-P3는 "챔피언 + 최선 신규센서(C12×2·C4×1)"로, C12는 EDA(E7)에서 **레짐 프록시**(전체 0.338 → 레짐 내 −0.02/−0.10)로 판명된 계열 — 랜덤 CV에선 중복·노이즈지만 H-T1의 주인공.

**실행 규칙 (남용 방지 — R10 연동)**:

1. **보고 전용 레인.** 트랙은 모델 선정(M5a)·튜닝(M5b)·제출물 확정 어디에도 개입하지 않는다 — G2 동결("피처 재변경 금지")은 그대로 유효하다. 트랙 성적은 벤치 표·REPORT에 병기만 한다.
2. **동일 조건 원칙.** 각 단계에서 트랙은 챔피언과 같은 fold 인덱스·같은 파라미터 규율로 평가한다. M5b에서는 재튜닝 금지 — 챔피언 확정 파라미터의 **전이 평가 1회**만.
3. **조회 규율.** 트랙의 valid/test 조회도 챔피언과 동일하게 마일스톤 종료 시 1회로 제한 (F-P3의 valid/test는 M5a에서 최초 기록).
4. **승격(챔피언 교체)은 예외 절차** — 다음 3조건 동시 충족 시에만 G2 재판정을 발의한다: ① 동일 조건 CV ΔCV ≤ −0.3 ② M7(Lot-CV·time-split) 지표에서도 우위 또는 동률 ③ fold별 std 감안 시 우연 아님. **valid/test 성적을 근거로 한 승격은 금지** (R10).
5. **은퇴 규칙.** M5a·M7·M8 전 구간에서 일관 열세면 M8 종료 후 트랙을 닫고, 최종 REPORT 비교표에 전 구간 성적을 기록으로 남긴다 ("계속 추적했고 끝까지 못 넘었다"는 것 자체가 코어 10 선정의 최종 증거가 된다).
6. **기록.** `results.json`의 각 실험 레코드에 `"feature_set": "F-C10"|"F-T15"|"F-P3"` 필드를 추가하고, REPORT에 트랙×단계 매트릭스(M5a/M5b-전이/M7/M8-S1/M8-S2)를 의무 수록한다.

### 9.4 시간 재분할 시뮬레이션 (M8) — "새 trace가 들어온다"의 최종 검증

**목적.** 대회 분할(동일 기간에서 wafer 랜덤 추출)은 "과거로 학습해 미래를 예측하는" 운영 상황을 대변하지 못한다. 전체 데이터를 병합해 **날짜 기준으로 train/valid/test를 새로 구성**하면, "지금까지 수집된 trace로 학습 → 이후 유입되는 trace를 예측"하는 상황을 그대로 모사할 수 있다. M7의 time-split(train 내부 80/20)이 값싼 조기 진단이라면, M8은 정답을 포함한 전 데이터로 수행하는 확장 최종판이다.

**방화벽 (가장 중요).** 이 절차는 valid/test 정답을 학습에 사용하는 **유일한** 절차다. 따라서 ① 대회 제출물(M0~M7 산출)이 전부 확정·저장된 **후에만** 실행하고, ② 별도 노트북(`modeling_v8_temporal_check.ipynb`)으로 격리하며, ③ 여기서 얻은 어떤 정보도 대회 트랙의 피처·모델·파라미터에 역유입하지 않는다 (R9).

**데이터 재구성 절차.**

1. 병합: train(11,939 WF) + valid_X×valid_Y_answer(1,990) + test_X×test_Y_answer(1,990) = **15,919 WF**. 병합 후 WF 수·중복 0 assert.
2. 대표시각: WF 첫 row의 C40. **Lot(C20) 단위로 묶어 "Lot 첫 wafer 시각" 기준으로 정렬** — 컷 경계에서 한 Lot이 두 분할에 걸치지 않게 보장 (Lot 스팬 ~22분이라 드물지만 규칙으로 강제; Lot 간접 누수 차단).
3. 분할: 누적 WF 수 기준 **new-train 70% / new-valid 15% / new-test 15%** 시간순 컷. 실행 시 실제 컷 날짜를 기록 (예상: new-test ≈ 캠페인 마지막 ~10일, post_pm_days 후기 구간).
4. 재현 규율: **학습 시점 산출물을 전부 new-train만으로 재생성** — pm_log 재탐지(new-train의 C33/C40만 사용), C23_te 재적합. 단 피처셋·모델·하이퍼파라미터는 대회 트랙 확정본을 **고정**(재탐색 금지, 재적합만) — M8은 "구성의 미래 견고성" 검증이지 새 최적화가 아니다. new-valid는 early stopping 전용.

|---|---|---|---|
| **S1 (필수, G5)** | 시간순 70/15/15 | 양 레짐을 모두 본 모델이 **미래 구간**(캠페인 후기)을 맞히는가 | 미래 test RMSE + 대회 test와의 갭 보고, 주별 오차 곡선 asset |
| S2 (진단) | new-train = PM 이전 전체(~12-23)만 → new-test = PM 직후 2주 | **한 번도 본 적 없는 레짐**이 오면 어떻게 되는가 (최악 시나리오) | 성능 하락을 정량화해 문서화 — "실패"가 아니라 **신규 레짐엔 재학습 필요**의 근거. **원본 `predict_prepm_only.py`가 동일 시나리오 구현 — 로직 재사용** + 원본 기준값 대조(raw RMSE ~700대·bias ≈ −486 → **R2R 보정 후 ~90**, COLD 분석). raw와 평균 bias 보정 후(R2R 모사) 두 수치를 함께 보고 |
| S3 (선택, D6) | expanding-window walk-forward (주 단위: 1~k주 학습 → k+1주 예측 반복) | 운영에서 **재학습 주기**를 얼마로 잡아야 하는가 | 주별 forward RMSE 곡선 → 재학습 주기 권고 도출 |

**모델 비교 포인트.** S1~S3를 최종 선정 모델과 **선형 변형(§8.3 후보 6)** 두 가지로 실행해 비교한다 — 트리는 관측 범위 밖 `days_since_last_pm`에서 상수로 수렴하고 선형(스플라인)은 추세를 연장하므로, 미래 구간에서 순위가 뒤집힐 수 있다. 뒤집히면 "대회 제출은 트리, 운영 권고는 하이브리드"로 트랙별 권고를 분리한다.

**피처셋 트랙 비교 포인트 (v1.6).** S1·S2는 **3트랙(F-C10·F-T15·F-P3, §8.5) 모두** 실행한다 (최종 선정 모델 기준; 선형 변형은 F-C10만 유지). 핵심 검정은 **S2×H-T1**: S2에서는 레짐 피처가 학습 구간에서 전부 상수라 모델이 레짐 점프를 볼 수 없는데, F-P3/F-T15의 C12 계열(레짐 프록시, C17 동형)은 X 채널로 새 레짐을 부분 감지할 수 있다 → **raw bias(−486 수준)가 F-C10보다 완만한지** 측정. 완만하면 "레짐 미기록/pm_log 공백 상황의 백업 피처셋"으로 운영 권고에 수록하고, 차이가 없으면 H-T1 기각을 기록한다 (어느 쪽이든 대회 트랙 제출물에는 영향 없음 — §8.5 규칙 1).

**증강 시나리오 전제 (v1.5).** 미관측 PM 거동을 합성할 때(스모크 A-4 확장·S-변형)는 팀 시뮬레이터 세계관을 따른다 — **quiet PM(클리닝 포함)은 현 레벨 유지, 레짐 전환은 loud에서만**. 구 가정("minor→저불량 복귀")으로 만든 합성 데이터와 혼용 금지. 이 전제 자체("quiet major ~70%", "클리닝=레벨 유지")는 **관측 n=0의 팀 전제**로 가정 대장에 기록하고, 운영에서 첫 후속 PM이 실제 관측되면 잔차 모니터링으로 판정·갱신한다.

**해석 주의.** new-test는 post-PM 후기 구간만 포함하므로 절대 RMSE를 대회 수치와 직접 비교하지 말 것. ① 같은 기간의 wafer만 추린 대회-분할 성능과 비교하거나 ② 레짐 피처 유/무 모델 간 상대 비교로 읽는다.

**산출물.** `outputs/temporal_results.json`(시나리오×모델×**피처셋 트랙** RMSE), 주별 열화 곡선 asset, H-T1 판정(S2 트랙별 raw bias 비교), REPORT의 **"신규 trace 대응" 절** — 운영 권고 3종: 재학습 주기(S3), 신뢰구간 확대 계수(미래 구간 오차 기반), 신규 PM 발생 시 절차(= pm_log 1줄 추가 + 재학습, §7.3 스모크 테스트로 증명).

## 10. Phase 6 — 오차분석·신뢰도·해석

1. **decile 편향 재측정 (G4)**: v5와 동일 방식으로 정답 10분위 RMSE·편향 표 + 그래프. 핵심 확인 = 742~976 구간 +58.8 해소 여부 (이 구간은 레짐 경계 레벨(pre 640↔post 1120 사이)이므로 레짐 피처로 잡혀야 정상). 양극단 수축(shrinkage)은 트리 구조 한계로 잔존 예상 — 필요시 OOF 기반 isotonic 보정을 후보로만 기록.
2. **그룹별 중요도 + 기여 분해**: gain을 {레짐/시간, 센서, 레시피} 그룹으로 합산해 비율 보고 (FS 구성 재현 확인) + **블록 ablation ΔCV와 M-T 대비 증분으로 실제 기여 분해** (§8.4 — gain 비율은 참고 지표, 기여 판정은 ablation). 3각 비교표(v5/M-T/v8) 수록.
3. **신뢰도 계층 재적용** (v5_confidence 자산 재사용): split conformal 90% 구간 + fold σ. 이번엔 **Lot 단위 OOF로 재보정**해 신규 Lot에 정직한 구간 제공 (세션4 권고 이행). 산출: `outputs/valid_confidence.csv`, `test_confidence.csv`.
4. **레시피별 오차**: C23_14/C23_12 (v5 최대 오차 레시피) 재확인.

### A-4. 스모크 테스트 (Phase 3 마지막 — v1.5 시나리오 교체: 신세계관 "quiet=현 레벨 유지" 반영)

> 구버전 기대값("가상 minor 추가 → `is_high_regime` 1→0 복귀, `high_regime_days`=0")은 **폐기** — 신세계관에선 quiet가 상태를 바꾸지 않는다. 아래 3종으로 교체.

**1단계 (피처 거동, P1 증명)**: pm_log 사본에 가상 PM을 각각 추가 → 피처 재계산 → 이벤트 이후 wafer들의 거동 assert:

| 시나리오 | 가상 이벤트 | 기대 피처 거동 |
|---|---|---|
| (a) 고불량 중 quiet PM | `{2019-01-20, minor, quiet}` | `is_high_regime` **1 유지**, `high_regime_days` **연속** (앵커=12-24 그대로 → 리셋 금지, ≈27일에서 계속 증가), `days_since_last_pm`만 리셋 |
| (b) 저불량 중 quiet PM | `{2018-12-15, major, quiet}` (조용한 major) | `is_high_regime` **0 유지** (type이 major여도 점프 예측 없음), `high_regime_days`=0 유지, dslp 리셋 |
| (c) 신규 loud PM | `{2019-01-20, major, loud}` | `is_high_regime` 상태 갱신, `high_regime_days` **0부터 재시작** (새 loud 앵커), dslp 리셋 |

**2단계 (예측 밴드 체크 — 시나리오별 오작동 실측)**: 시나리오별 합성 피처 벡터를 학습된 모델에 넣어 예측 밴드를 확인한다.

- **(a) 고불량 중 quiet PM** (`is_high=1`·hrd≈27~40 연속·dslp≈0~수일·C33≈1~3·대표 센서값): 예측이 **현 감쇠 레벨 밴드(≈950~1,100) 유지**인가. 실패 모드 **두 방향 모두** 판정 — ~638 복귀(구세계관 오작동)도, ~1400 스파이크(hrd 리셋 버그 — v1.5 앵커 수정이 막는 대상)도 실패. 특히 C33은 verdict 무관 모든 PM에서 리셋되는 장비 카운터라(수정 불가, R12) "is_high=1 ∧ C33≈1 ∧ hrd≈27"은 미학습 조합 — feature_fraction(0.632)으로 hrd를 못 본 트리들이 C33 프록시로 **초기-major(~1400) 방향으로 끄는 잔여 오염**을 여기서 실측한다.
- **(b) 저불량 중 quiet PM** (`is_high=0`·dslp≈0~수일·C33≈1~3): 예측이 **저불량 밴드(≈550~750) 유지**인가. 기존 "is_high=0 ∧ C33<3" 미관측 조합 체크(저불량 관측은 C33≥3)를 이 시나리오로 재활용.
- **(c) 신규 loud PM** (`is_high` 갱신·hrd=0 재시작): **학습된 유일한 요란 패턴(~1400→감쇠)으로 라우팅**되는지 확인. 단 새 loud의 실제 레벨·방향(상승/하강)은 미관측 — 이 출력은 "모델이 아는 패턴"일 뿐이며 레벨 정합은 R2R·재학습 몫(§9.4 S2와 동일 구조). 운영 권고: 신규 loud 직후 신뢰구간 확대 + 우선 실검사.

(전 시나리오 모델 학습 미사용, 검증 전용 셀. 밴드 이탈 시 M8 증강 시뮬레이션에서 정밀 진단. 합성 데이터 생성 전제는 §9.4 "증강 시나리오 전제" 준수 — 구/신 세계관 혼용 금지.)
