## 0. 무엇을 보는가

M0b가 **WF 단위 집계 10피처**로 CV 40.35를 확정했고, M1·M2는 각각 C23_te·센서 풀 재선별을 얹었지만 코어 10을 못 넘었다. M3는 **예측 단위 자체**를 바꾼다 — WF로 뭉치기 전에 **행(step) 단위**로 C65를 예측하고 나서 WF 평균으로 되돌린다. 이것이 v5의 원래 프레임이었고(당시 valid 61.4), 질문은 하나다: **레짐 피처를 각 행에 broadcast한 위에서도, 행 단위 step×센서 신호가 WF 집계보다 나은가?**

PLAN의 사전 실험 **E10**은 이미 답을 낮춰뒀다 — v5 잔차와 레짐 피처의 상관이 ≈0이라, "v5(센서 프록시)가 레짐을 이미 흡수했다"면 역으로 "레짐을 직접 모델링하면 row-level이 더 보탤 게 없다"는 뜻. M3는 그 예측을 **저비용 확인 실험**으로 검증한다.

---

## 1. 방법 — v5 프레임 + 그룹 A/B broadcast, M0b와 동일 분할

- **행 피처(157개)**: v5 빌더 이식 — 원본 센서(행값) + **WF 전역 context**(센서별 mean/std/min/max broadcast) + `wf_nrows` + `row_pos`(WF 내 순번) + `C6`/`C7` LightGBM 범주형 + `C23` OOF 타깃인코딩(m=20, fold-내 train만 fit).
- **그룹 A broadcast**: `is_high_regime, high_regime_days, days_since_last_pm, C33, dslp_x_hour, hour, hour_x_c33`(WF 상수)를 각 행에 붙임. `C33` 등 원본과 겹치는 열은 병합 전 제거해 충돌 방지.
- **행별 `hour_row`**: 그룹 A의 `hour`(WF-first)와 별개로, 각 행 `C40` 타임스탬프의 시각 — row-level이 실제로 쓸 수 있는 유일한 행 단위 시간 신호.
- **그룹 B**(`C59/C60_mean_step4`)는 context 집계 안에 자연 포함.
- **공정 비교**: 행→WF fold 매핑을 M0b와 **동일한** `KFold(5, shuffle, 42)` WF 분할로 만들어, OOF를 행→WF 평균한 뒤 **WF-RMSE**를 낸다 → M0b와 ΔCV 직접 비교.
- **파라미터**: v5 row-level 레시피(`num_leaves=127, lr=0.03, feature_fraction=0.7, bagging=0.8`, 4000 rounds early stopping) — 행 단위에 맞춘 설정으로, 프레임에 유리하게 두고 판정.

## 2. 결과

| 지표 | **M3** (row-level) | M0b (WF-level, 코어10) | Δ (M3−M0b) |
|------|:---:|:---:|:---:|
| **CV(wafer, OOF)** | **50.868** | 40.347 | **+10.521** |
| valid | 49.273 | 38.399 | +10.874 |
| test | 48.875 | 38.912 | +9.963 |

fold별 `best_iter` = 970 / 347 / 758 / 807 / 570 — 조기 수렴/과소적합 아님(정상 학습). 즉 **50.87은 실력값**이지 예산 부족 산물이 아니다.

**판정 게이트**: ΔCV = +10.521 (≤ −0.3 이어야 채택) → **❌ 기각.** WF-level(M0b 코어 10) 프레임 확정.

## 3. 삼각 비교 — 레짐 broadcast의 효과와 한계

| 구성 | 프레임 | 레짐 피처 | valid RMSE |
|------|--------|:---:|:---:|
| v5 | row-level | ✗ (센서 프록시만) | 61.38 |
| **M3** | row-level | ✓ broadcast | **49.27** |
| M-T | WF-level | ✓ (시간 7만) | 44.62 |
| **M0b** | WF-level | ✓ + 센서 3 | **38.40** |

두 가지가 동시에 보인다.

첫째, **레짐 broadcast는 row-level 프레임을 크게 끌어올린다** — v5 61.4 → M3 49.3, 약 −12pt. 레짐 신호 자체는 강력하다는 재확인.

둘째, **그럼에도 WF-level에 한참 못 미친다** — 같은 레짐을 넣어도 row-level 49.3 vs WF-level 38.4로 ~11pt 벌어진다. 원인은 타깃 구조에 있다: **C65는 WF 내 상수**(한 WF의 모든 행이 같은 라벨)라, 행 단위 예측은 같은 정답을 여러 행에 나눠 학습하며 within-WF 잡음을 끌어들인다. 정작 유효 신호(레짐 + `C59/C60_mean_step4` 같은 **step-pivot 집계**)는 본질적으로 WF 속성이라, WF-level 테이블이 이를 **정밀하게** 인코딩하는 반면 row-level의 context broadcast(mean/std/min/max)는 그걸 **거칠게 재구성**할 뿐이다. 범주형 `C7`×센서 상호작용도 그 손실을 못 메운다. 예측을 다시 WF 평균으로 되돌리는 단계는 신호 없이 분산만 더한다.

## 4. 해석 — E10 예측의 정량 확증

E10은 "상관 ≈0 → row-level 첨가 기대 하향"이라 했다. 결과는 그보다 **강하다**: row-level은 단지 "무이득"이 아니라 **능동적으로 열세**(+10.5pt)다. 레짐을 직접 모델링한 순간(M0b), 행 단위 프레임은 WF 집계가 이미 깔끔히 쥔 신호를 희석시킬 뿐이다. **M0 교체가 주경로**라던 PLAN의 판단이 수치로 확정됐다.

M1(+0.68)·M2(+3.59)·M3(+10.52)가 **모두 코어 10을 못 넘었다** → v8의 피처·프레임 탐색은 여기서 **코어 10 WF-level로 수렴**한다. 남은 개선 여지는 피처가 아니라 **모델·앙상블·정직성 평가**(M4~M8) 쪽.

## 5. 검증 메모 (클라우드 미러)

- 피처 파이프라인이 원본 M0b를 **정확 재현**(CV 40.347 = README 40.35) → 빌더 정합 확인.
- 행 빌더 무오류(157피처 × 123,614행). 초기 발견한 **`C33` 컬럼 충돌**(raw C33 vs 그룹 A broadcast C33)은 병합 전 원본열 제거로 수정 완료.
- 5-fold 학습→WF 집계→게이트 전 경로 무오류 실행. 45초 실행 제한상 미러는 **축소 예산**으로 방향만 확인(M3 53.6 vs M0b 49.4, 동일 예산 → 동일 기각), **정식 수치는 로컬 정식 실행**(M3 50.868)에서 확정.
