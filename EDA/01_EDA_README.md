# 01_EDA.ipynb 안내서

## 이 노트북은 무엇을 하는 건가요?

반도체 제조 공정에서 수집된 센서 데이터를 **처음으로 살펴보는** 탐색적 데이터 분석(EDA) 노트북입니다.

쉽게 비유하면, 요리를 시작하기 전에 재료의 상태를 확인하는 과정과 같습니다.
- 재료(데이터)가 몇 개인지
- 상한 재료(결측값, 이상치)는 없는지
- 어떤 재료가 맛(결함 수치)에 영향을 주는지

이런 것들을 확인하고, 이후 **예측 모델을 만들 때 어떤 전략**을 써야 할지 방향을 잡는 것이 목적입니다.

---

## 배경 지식

### 데이터가 뭔가요?

SK하이닉스 반도체 공장에서 웨이퍼(Wafer)를 만들 때, 장비에 달린 센서들이 온도, 압력, 시간 등 다양한 수치를 기록합니다. 이것을 **FDC(Fault Detection and Classification) 데이터**라고 합니다.

- **웨이퍼(WF)**: 반도체 칩을 만드는 둥근 원판. 하나의 웨이퍼에서 여러 개의 반도체 칩이 만들어집니다.
- **C64**: 각 웨이퍼를 구분하는 고유 ID (예: `C64_1`, `C64_2`, ...)
- **C65**: 우리가 **예측하고 싶은 값** (결함 테스트 수치). 이 숫자가 높으면 결함이 많다는 뜻입니다.
- **C1 ~ C63**: 장비 센서에서 측정된 다양한 값들 (온도, 압력, 시간 등)

### 목표

센서 데이터(C1~C63)를 보고 → 결함 수치(C65)를 **미리 예측**하는 모델을 만드는 것입니다.

## 노트북 구성 (셀별 설명)

### 셀 0: 데이터 불러오기

5개의 CSV 파일을 불러옵니다.

| 파일 | 설명 | 용도 |
|------|------|------|
| `train_data.csv` | 학습용 데이터 (센서값 + 정답) | 모델이 패턴을 배우는 데 사용 |
| `valid_X.csv` | 검증용 데이터 (센서값만) | 모델 성능을 중간 점검할 때 사용 |
| `test_X.csv` | 테스트용 데이터 (센서값만) | 최종 성능을 평가할 때 사용 |
| `valid_Y_answer.csv` | 검증용 정답 | 중간 점검의 채점표 |
| `test_Y_answer.csv` | 테스트용 정답 | 최종 평가의 채점표 |

또한 유닉스 타임스탬프(숫자로 된 시간)를 사람이 읽을 수 있는 날짜/시간 형식으로 변환합니다.

### 셀 1: 컬럼(열) 기본 정보 파악

68개 컬럼 각각에 대해 다음을 확인합니다:
- **데이터 타입**: 숫자인지, 문자인지
- **고유값 개수**: 값이 얼마나 다양한지
- **결측값(빈칸) 비율**: 데이터가 비어있는 비율
- **상수 컬럼**: 모든 행의 값이 같아서 분석에 쓸모없는 컬럼

### 셀 2: 결측값(빈칸) 패턴 분석

빈칸이 있는 컬럼들을 찾고, 빈칸이 **어떤 패턴**으로 나타나는지 히트맵(색깔 지도)으로 시각화합니다.
- 빈칸이 랜덤하게 나타나면 → 평균값 등으로 채우기
- 빈칸이 특정 패턴으로 나타나면 → "빈칸 여부" 자체를 새로운 정보로 활용

### 셀 3: 웨이퍼(WF) 그룹 구조 확인

하나의 웨이퍼가 데이터에서 몇 행(row)을 차지하는지 확인합니다.
- 웨이퍼마다 여러 공정 단계를 거치므로, 한 웨이퍼에 여러 행이 있습니다.
- C65(결함 수치)가 같은 웨이퍼 내에서 항상 동일한 값인지도 검증합니다.

### 셀 4: 예측 대상(C65) 분포 분석

C65 값이 어떤 모양으로 분포하는지 4가지 차트로 확인합니다:
1. **히스토그램 + KDE**: 값들이 어디에 몰려있는지 (종 모양인지, 한쪽으로 치우쳤는지)
2. **박스플롯**: 중앙값, 사분위 범위, 극단값을 한눈에 비교
3. **QQ-플롯**: 정규분포(종 모양)와 얼마나 비슷한지 확인
4. **기초 통계표**: 평균, 표준편차, 최솟값, 최댓값 등 숫자 요약

### 셀 5: 시간에 따른 변화 분석 (Temporal Drift)

시간이 흐르면서 C65 값이 변하는 경향이 있는지 확인합니다.
- 만약 시간이 지나면서 결함이 늘어나는 추세라면, "시간"도 중요한 예측 변수가 됩니다.
- Train/Valid/Test 데이터가 시간적으로 겹치는지도 확인합니다.

### 셀 6: 데이터 분포 변화 확인 (Covariate Shift)

Train 데이터와 Valid/Test 데이터의 **센서값 분포가 비슷한지** 확인합니다.

비유하면, 학습할 때 본 문제와 시험 문제의 난이도/유형이 비슷한지 확인하는 것입니다.
- **Adversarial Validation**: "이 데이터가 Train인지 Test인지 구분할 수 있느냐"를 테스트
- AUC 0.5에 가까우면 → 구분 불가 = 분포가 비슷함 (좋은 신호)
- AUC 1.0에 가까우면 → 완전히 구분됨 = 분포가 다름 (주의 필요)

### 셀 7: 센서값과 결함의 상관관계

어떤 센서값(C1~C63)이 결함 수치(C65)와 **관련이 깊은지** 두 가지 방법으로 측정합니다:
- **피어슨 상관계수**: 직선적인 관계의 강도 (-1 ~ +1)
- **스피어만 상관계수**: 순위 기반 관계의 강도 (비선형 관계도 감지)

또한 센서값끼리 서로 너무 비슷한 정보를 담고 있는지(**다중공선성**) 히트맵으로 확인합니다.

### 셀 8: PM(C33) 영향 분석

C33은 **PM(Preventive Maintenance, 예방 정비)** 이후 경과 시간을 나타냅니다.
- 장비를 정비한 직후 vs 오래 지난 후, 결함 수치에 차이가 있는지 확인합니다.
- PM 주기가 결함에 영향을 준다면, 이를 예측에 활용할 수 있습니다.

### 셀 9: 이상치(극단값) 탐지

정상 범위를 벗어나는 **극단적인 값**이 얼마나 있는지 확인합니다.
- **IQR 방법**: 데이터의 중간 50% 범위를 기준으로, 그 1.5배 이상 벗어난 값을 이상치로 판단
- C65(결함 수치)와 주요 센서 피처 모두에 대해 이상치 비율을 계산합니다.

### 셀 10: 범주형 변수와 결함의 관계

숫자가 아닌 **범주(카테고리)** 변수가 결함에 미치는 영향을 분석합니다:
- **C6 (Recipe)**: 제조 레시피(조리법과 비슷한 개념). 레시피에 따라 결함 수치가 달라지는지
- **C7 (Step)**: 공정 단계. 어떤 단계에서 결함 차이가 나는지
- **C6 x C7 조합**: 레시피와 단계의 조합별 결함 패턴

## 사용된 주요 분석 기법 용어 정리

| 용어 | 쉬운 설명 |
|------|----------|
| EDA | 데이터를 처음 만졌을 때 전반적으로 살펴보는 과정 |
| 히스토그램 | 값의 빈도를 막대로 보여주는 차트 |
| KDE | 히스토그램을 부드러운 곡선으로 표현한 것 |
| 박스플롯 | 데이터의 분포를 상자와 수염으로 요약한 차트 |
| QQ-플롯 | 데이터가 정규분포(종 모양)와 얼마나 비슷한지 보는 차트 |
| 상관계수 | 두 변수의 관련성 강도 (-1~+1, 0이면 무관) |
| 다중공선성 | 여러 변수가 너무 비슷한 정보를 담고 있는 문제 |
| Adversarial Validation | Train/Test 분포 차이를 확인하는 기법 |
| IQR | 데이터 중간 50%의 범위, 이상치 판단 기준 |
| Skewness (왜도) | 분포가 한쪽으로 치우친 정도 |
| Kurtosis (첨도) | 분포의 꼬리가 두꺼운 정도 |

## 이 분석 이후에는?

이 EDA 결과를 바탕으로 다음 단계를 진행합니다:
1. **전처리**: 쓸모없는 컬럼 제거, 결측값 처리, 이상치 처리
2. **피처 엔지니어링**: 새로운 변수 생성 (예: PM 경과 시간 관련 파생 변수)
3. **모델링**: 머신러닝 모델을 학습시켜 C65를 예측
4. **평가**: Valid/Test 데이터로 예측 성능 측정
